I have a small private dread now when I ask an agent to "add support for X".

Not because it will fail. Failure would be clean. It is usually worse than that: it succeeds in the most professionally unsettling way. It finds the files, follows the conventions, adds tests, updates the wiring, and hands back a diff with the calm confidence of a person who has never had to maintain anything. The code compiles. The tests pass. The shape is slightly wrong.

The provider vocabulary has leaked into the domain model. A lifecycle has acquired a new state because the external API had one. A fallback has appeared where a configuration error should have been fatal. A helper has been abstracted two commits before it has earned the right to exist.

None of these are hallucinations in the usual sense. The model did not invent a non-existent method or cite a fake package. It did something more ordinary and more dangerous: it expanded the wrong form.

This is the bit of LLM-assisted programming I think we are still naming badly. We talk about prompts, specifications, agents, context windows, evals, autonomy. All useful words, in their places. But they mostly circle the machinery. The more important unit is the shape the human is trying to preserve while the machinery expands it.

LLMs made code generation cheap. They did not make system judgement cheap.

## The thing being held

By "shape" I do not mean architecture in the grand diagram sense. I mean the working mental form of the change: what kind of thing it is, where it belongs, which names are allowed to cross which boundary, which invariants are load-bearing, and which tempting moves are explicitly not part of this slice.

A spec tells you what behaviour should exist. A shape tells you what kind of implementation would be faithful to the system.

Those are not the same thing.

"Add Stripe support" is a task. "Add Stripe behind the existing payment provider interface, map Stripe events into our current order transitions, store provider event IDs for idempotency, and do not let Stripe status names enter the order domain" is a shape. It constrains the expansion. It tells the model what must remain itself.

The negative space is doing as much work as the positive instruction. No new lifecycle. No second source of truth. No magic fallback. No provider vocabulary in the domain. No general payment orchestration framework because we are adding the second provider and the model has read enough blog posts to know what comes next.

This is why a short instruction from someone who understands the system can be more effective than a long document generated by someone who does not. The length is not the point. The load-bearing thing is whether the instruction preserves the form.

## Expansion is not judgement

The model is extremely good at expansion. That is the source of both the power and the danger.

Give it a partially specified idea and it can elaborate. Files, types, tests, migrations, fixtures, error handling, documentation, a tidy implementation plan, a confident summary. It can make the idea look more complete than it is. This is useful when the idea is well-shaped. It is ruinous when the idea is vague, because the vagueness does not remain small. It blooms.

Before these tools, an under-shaped idea often failed by being slow. You tried to implement it, got stuck, noticed the boundary was wrong, had an unpleasant afternoon, and learned something. The friction was annoying, but it was also diagnostic. It told you where you had not thought.

Now the under-shaped idea can become a large coherent diff before the discomfort has time to appear. The agent does not pause at the moment where you would have felt resistance. It supplies a plausible answer and moves on. By the time you see the result, you are not reviewing your intent in code. You are reverse-engineering a foreign design that happens to be written in your house style.

That is the failure mode. Not "the model wrote bad code", although it may have. The deeper problem is that code production outran shape formation.

## Review by recognition

There is a particular feeling when an LLM-generated change is going well. You look at the diff and recognise it.

Not in the superficial sense that the names are familiar or the framework patterns are correct. You recognise the move. The adapter is where you expected it. The translation happens at the boundary. The domain model did not absorb the provider's anxieties. The tests assert the invariant rather than the implementation accident. The change is larger than you would have typed by hand in that amount of time, but it is still recognisably yours.

That is the good loop:

1. The human forms the shape, often with help.
2. The model expands it.
3. The human steers the expansion while it is still small.
4. The human reviews by recognition.

The third step matters more than people like to admit. The safest time to correct an agent is before the wrong shape has acquired supporting code. "This belongs in the adapter, not the service" is cheap early. Later it is an afternoon. Later still it is institutional knowledge with a sad comment above it.

This is also why large generated pull requests are not inherently bad, but unrecognisable ones are. A two-thousand-line diff that is a mechanical expansion of a shape I already hold can be easy to review. A two-hundred-line diff that introduces a new conceptual boundary I did not ask for can be impossible to review responsibly.

Line count is a poor proxy. Recognisability is the signal.

The useful question is not "how much did the agent produce?" It is "who is holding the shape, and can they still see it in the result?"

## Where to put the friction

The standard productivity fantasy is that friction is the enemy. Remove enough of it and software gets faster. This is half right in the usual dangerous way.

Typing boilerplate is friction. Waiting for a scaffold is friction. Looking up the exact incantation for a framework API is friction. Fine. Remove it.

But some friction is where judgement enters the system. Database migrations. Permission changes. New dependencies. Lifecycle states. Background jobs. Fallback behaviour. Security boundaries. Cross-domain coupling. New abstractions with names that sound as if they have always existed.

These are not merely implementation details. They are places where the system can change shape while everyone is looking at the code.

An agent should be fast around them, not through them. It should surface them. It should ask. It should stop with a proposed shape before expanding into twenty files. And the human should have enough taste, context, and nerve to say the boring thing: no, that is not what this is.

That may sound like a conservative posture. It is not, really. It is the posture that lets you move quickly without handing the system's form over to whatever pattern happened to be statistically nearby.

## A narrow claim

I am not arguing that humans should lovingly handcraft every line while the model watches in silence. That would be absurd, and also not how I work. I use agents constantly. I ask them to explore codebases, interrogate third-party APIs, find edge cases, produce drafts, write tests, and implement slices I would rather not type myself.

The narrow claim is this: the human role has moved up a level, but it has not disappeared. The human is less often the typist and more often the holder of shape. If that sounds grandiose, call it something duller: someone has to know what the thing is supposed to become.

The model can help you discover that. It can ask good questions. It can bring back constraints from the codebase. It can notice conventions you missed. It can give you three possible boundaries and explain their trade-offs. But it cannot be the owner of the shape, because ownership is not the same as fluency. Ownership includes caring what happens to the system after the diff is merged.

This is the part that gets obscured when we talk too much about autonomy. The question is not how far the agent can run. The question is whether the human can still recognise the place it has run to.

## The maxim

Shape-driven LLM programming is not a methodology in any interesting ceremonial sense. It is a refusal to confuse expansion with judgement.

Use the model to investigate. Use it to draft. Use it to implement. Use it to test your own understanding. But do not let the generated artefacts become the place where the system's form is decided by accident.

The human holds the shape. The model expands it. The human steers while the expansion is still cheap to correct. The final review is not archaeology; it is recognition.

When that loop holds, LLM-assisted programming feels almost unfair. When it breaks, the codebase receives perfectly plausible answers to questions nobody quite meant to ask.

The scarce thing is not code. The scarce thing is knowing what shape the code is allowed to take.
